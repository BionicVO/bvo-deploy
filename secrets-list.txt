BionicVO — Secret Manager secrets and where each is referenced
================================================================
Create each of these with:
  gcloud secrets create <secret-name> --replication-policy=automatic
  printf '<value>' | gcloud secrets versions add <secret-name> --data-file=-

Format below: secret name -> secretKeyRef.name (Cloud Run manifests) / env
var it's injected as, and which file(s) reference it.

STAGING
-------
bvo-db-host-staging       -> secretKeyRef: bvo-db-host-staging       -> env DB_HOST
                             used in: backend-service.staging.yaml, cloudbuild-backend.yaml
bvo-db-port-staging       -> secretKeyRef: bvo-db-port-staging       -> env DB_PORT
                             used in: backend-service.staging.yaml, cloudbuild-backend.yaml
bvo-db-username-staging   -> secretKeyRef: bvo-db-username-staging   -> env DB_USERNAME
                             used in: backend-service.staging.yaml, cloudbuild-backend.yaml
bvo-db-password-staging   -> secretKeyRef: bvo-db-password-staging   -> env DB_PASSWORD
                             used in: backend-service.staging.yaml, cloudbuild-backend.yaml
bvo-db-name-staging       -> secretKeyRef: bvo-db-name-staging       -> env DB_NAME
                             used in: backend-service.staging.yaml, cloudbuild-backend.yaml
bvo-redis-url-staging     -> secretKeyRef: bvo-redis-url-staging     -> env REDIS_URL
                             used in: backend-service.staging.yaml
bvo-elevenlabs-key-staging -> secretKeyRef: bvo-elevenlabs-key-staging -> env ELEVENLABS_API_KEY
                             used in: backend-service.staging.yaml
                             REQUIRED — ElevenLabsClientService.js constructs the
                             SDK client at import time; missing this crashes the
                             whole process before it can bind to PORT.
bvo-jwt-secret-staging     -> secretKeyRef: bvo-jwt-secret-staging     -> env SECRET
                             used in: backend-service.staging.yaml
                             REQUIRED — JwtService signs/verifies every auth token
                             with SERVER_CONFIG.SECRET (process.env.SECRET). Also
                             used to sign cookies (cookieParser(SERVER_CONFIG.SECRET)
                             in bootstrap.ts). Missing this makes login fail with a
                             400 "secretOrPrivateKey must have a value" error.

PRODUCTION
----------
bvo-db-host-prod       -> secretKeyRef: bvo-db-host-prod       -> env DB_HOST
                          used in: backend-service.prod.yaml, promote-to-prod.yaml
bvo-db-port-prod       -> secretKeyRef: bvo-db-port-prod       -> env DB_PORT
                          used in: backend-service.prod.yaml, promote-to-prod.yaml
bvo-db-username-prod   -> secretKeyRef: bvo-db-username-prod   -> env DB_USERNAME
                          used in: backend-service.prod.yaml, promote-to-prod.yaml
bvo-db-password-prod   -> secretKeyRef: bvo-db-password-prod   -> env DB_PASSWORD
                          used in: backend-service.prod.yaml, promote-to-prod.yaml
bvo-db-name-prod       -> secretKeyRef: bvo-db-name-prod       -> env DB_NAME
                          used in: backend-service.prod.yaml, promote-to-prod.yaml
bvo-redis-url-prod     -> secretKeyRef: bvo-redis-url-prod     -> env REDIS_URL
                          used in: backend-service.prod.yaml
bvo-elevenlabs-key-prod -> secretKeyRef: bvo-elevenlabs-key-prod -> env ELEVENLABS_API_KEY
                          used in: backend-service.prod.yaml
                          REQUIRED — same reason as staging's; separate key per
                          environment (not shared with staging).
bvo-jwt-secret-prod     -> secretKeyRef: bvo-jwt-secret-prod     -> env SECRET
                          used in: backend-service.prod.yaml
                          REQUIRED — same reason as staging's. Separate signing
                          secret per environment: rotating/compromising staging's
                          never invalidates prod sessions or vice versa.

Notes
-----
- Secret name and secretKeyRef.name are always identical in this setup — the
  Cloud Run manifests reference the secret by its own name, just mapped to a
  different env var name inside the container.
- key: "latest" everywhere, meaning each secretKeyRef always pulls the newest
  version of that secret — rotate by adding a new version, no manifest change needed.
- 16 secrets total: 8 per environment x 2 environments (staging, prod).
  Stripe keys are intentionally excluded — handled inside the app itself,
  not via Secret Manager/Cloud Run env vars.
- ElevenLabs is required at startup (see above) — wired in.
- JWT secret (env SECRET) is required at request-time (any login/token
  verify) — wired in. Generate with openssl rand, piped directly into
  Secret Manager so the raw value never appears in chat/shell history:
    openssl rand -base64 48 | gcloud secrets create bvo-jwt-secret-staging \
      --replication-policy=automatic --data-file=-
    openssl rand -base64 48 | gcloud secrets create bvo-jwt-secret-prod \
      --replication-policy=automatic --data-file=-
  Also remember: CLIENT_URLS (bootstrap.ts CORS allowlist) is now set as a
  plain env var directly in each backend manifest (not a secret) — must
  match the frontend's exact Cloud Run origin per environment.
- SendGrid and GCS credentials are NOT wired in, and don't need to be:
  both are read lazily (SendGrid inside SendgridClient.sendEmail(), GCS
  settings pulled from a DB-backed admin config inside GCSClientService),
  so a missing value only throws a normal, catchable error when actually
  used — it doesn't crash the container at startup like ElevenLabs did.
  Wire them in later only if you want CI-managed secrets instead of
  whatever config path the app currently falls back to.
